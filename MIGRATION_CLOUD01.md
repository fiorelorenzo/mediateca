# Migration plan: `vps-jellyfin` (CPX22 FSN1) Ôćĺ `cloud01` (Hetzner Dedicated, HEL1)

Goal: move the entire Servarr/Jellyfin stack from the current Hetzner CPX22
(2 vCPU AMD shared, 4 GB, FSN1 Falkenstein) to a Hetzner Server Auction
**dedicated** machine (Xeon E3-1275v6, 4c/8t Kaby Lake, 64 GB DDR4 ECC,
2├Ś 512 GB NVMe in mdadm RAID 1, HEL1 Helsinki) and add Tdarr for
asynchronous transcoding. The current Storage Box (BX11, FSN1) is being
**replaced** by a new BX11 at HEL1 to keep CIFS intra-DC. Quality profiles
already cap at 1080p so Tdarr can run a simple idempotent encoding flow
without 4K-aware logic.

**Storage architecture** (chosen 2026-04-30):
- Media (Ôëł100 GB today) lives on the new HEL1 Storage Box, mounted via CIFS
  on the dedicated as `/mnt/storagebox` exactly like today.
- Tdarr **transcode cache** lives on the local NVMe RAID 1 (`/var/lib/tdarr-cache`).
  This is what allows fast encoding without saturating CIFS.
- Tdarr **reads** the source from CIFS, **encodes** locally, **writes back**
  the result to CIFS via Tdarr's "Replace Original File" plugin (atomic
  rename ÔÇö file remains continuously readable by Jellyfin during encode,
  unavailable for ~1-2 s during the final swap).

This is a **runbook**: follow phase-by-phase, tick each checkbox, and stop
on any unexpected output. Rollback instructions are at the bottom.

Estimated wall-clock: ~3 hours active (mostly waiting on rescue/installimage
+ media pre-sync), ~25 minutes of public downtime.

### 2.1 Add Tdarr to `docker-compose.yml`

Append this service block near the bottom (after `headscale`):

```yaml
  # =========================================
  # Tdarr ÔÇö asynchronous transcoding pipeline.
  # Re-encodes new imports to H.264 1080p + AAC stereo + AC3 5.1 so
  # Jellyfin can serve them as direct-play to almost any client. Quality
  # profiles cap at 1080p, so Tdarr never sees 4K input.
  # 4 cores / 8 threads Ôćĺ 2 parallel FFmpeg workers without saturating
  # the host (leaves 2 threads for Jellyfin live-transcode + everything else).
  # =========================================
  tdarr:
    image: ghcr.io/haveagitgat/tdarr:latest
    container_name: tdarr
    restart: unless-stopped
    cpus: 4.0
    mem_limit: 12g
    environment:
      - TZ=Europe/Rome
      - PUID=${PUID}
      - PGID=${PGID}
      - serverIP=0.0.0.0
      - serverPort=8266
      - webUIPort=8265
      - internalNode=true
      - inContainer=true
      - nodeName=tdarr-cloud01
      - ffmpegVersion=7
    volumes:
      - ./config/tdarr/server:/app/server
      - ./config/tdarr/configs:/app/configs
      - ./config/tdarr/logs:/app/logs
      # Source / output: media lives on the HEL1 Storage Box, mounted via
      # CIFS on the host. Read+write happen here, but only the FINAL atomic
      # rename hits CIFS ÔÇö encoding writes go to /temp on local NVMe.
      - /mnt/storagebox/data/media:/media
      # Encode cache on local NVMe RAID 1 (~480 GB usable). Critical for
      # I/O perf: SMB roundtrip per write would tank libx264 throughput.
      - /var/lib/tdarr-cache:/temp
    networks:
      - servarr
```

### 3.2 Boot into rescue + run `installimage` with RAID 1

In Robot UI: select the new server Ôćĺ **Rescue** tab Ôćĺ activate
"Linux x64", confirm. Robot generates a one-time root password ÔÇö save as
`RESCUE_PASSWORD`. Reboot the server (Robot Ôćĺ Reset Ôćĺ "Execute an
automatic hardware reset").

Wait ~60 s for it to come back up in rescue mode, then SSH in:

```sh
ssh -o StrictHostKeyChecking=no root@${NEW_IP}
# Use RESCUE_PASSWORD (or your SSH key if pre-uploaded)
```

In the rescue shell, run `installimage`. It launches an interactive editor;
either `Esc` then paste this config, or write it via heredoc and pass it
via `-c`:

```sh
cat > /tmp/install.conf <<'CONF'
HOSTNAME cloud01
DRIVE1 /dev/nvme0n1
DRIVE2 /dev/nvme1n1
SWRAID 1
SWRAIDLEVEL 1
PART /boot ext3   1G
PART swap  swap   8G
PART /     ext4   all
IMAGE /root/.oldroot/nfs/install/../images/Ubuntu-2404-noble-amd64-base.tar.gz
CONF

installimage -c /tmp/install.conf
```

Watch for:
- `mdadm` creating `md0` (boot), `md1` (swap), `md2` (root) ÔÇö both NVMe
  drives must show `active sync` in both arrays.
- Any "ERROR" line. If it fails, common causes: NVMe device names differ
  (some boards expose `/dev/nvme2n1` as the second drive ÔÇö adjust DRIVE2),
  or the rescue image isn't fresh (re-activate rescue and reboot).

Once it prints `Installation complete`, reboot:
```sh
reboot
```

Wait ~60 s, then SSH back in (this time as `root` with **your SSH key** ÔÇö
installimage propagates the key from the rescue auth context):

```sh
ssh root@${NEW_IP} 'cat /proc/mdstat'
# Expected: 3 md devices, all "[UU]" (both disks in sync)
```

- [ ] `cat /proc/mdstat` shows md0/md1/md2 with `[UU]` on both disks
- [ ] `lsblk` shows nvme0n1 + nvme1n1 each partitioned identically

### 4.2 Rsync `/opt/servarr/` to `cloud01`

The cleanest path is a two-step rsync via your laptop (avoids public IPÔćĺIP rsync
permissions issues):

```sh
# Pull from old to local
rsync -avh --delete --exclude='caddy/data/' --exclude='caddy/config/' \
```

We **exclude** `caddy/data/` and `caddy/config/` on purpose: Caddy's ACME
state (account keys, issued certs) is fine to discard and re-issue. With ~9
subdomains and Let's Encrypt's rate limit of 50 certs/week per registered

## Appendix A: Why dedicated x86 (and not CAX41 cloud ARM)

Original plan was Hetzner CAX41 (16 ARM Ampere cores, ÔéČ38.42/mo cloud) but
ARM capacity was unavailable in any EU location during the migration
window (2026-04-30). Server Auction Xeon E3-1275v6 was picked as the next
best option:

- **Same price point** (ÔéČ40.70/mo dedicated vs ÔéČ38.42/mo cloud)
- **Hardware dedicated** ÔÇö no noisy-neighbor on Tdarr sustained load
- **64 GB ECC RAM** vs 32 GB on CAX41 ÔÇö overkill but free margin
- **Truly unmetered traffic** on 1 Gbit/s NIC (Hetzner dedicated T&C) vs
  20 TB/mo soft cap on cloud
- **Single-thread perf actually higher** (Kaby Lake @ 4.2 GHz boost vs
  Ampere Altra @ 3.0 GHz) Ôćĺ Jellyfin live transcode benefits
- **Multi-thread perf lower**: 4c/8t vs 16c Ôćĺ Tdarr backlog ~24h on 100 GB
  vs ~16h on CAX41. Acceptable since steady state is <1 file/day

Trade-offs accepted: setup fee (~ÔéČ39 one-time), no API provisioning (must
order via web), kernel updates require manual reboot via Robot UI.

If ARM capacity returns, a future migration to CAX41 is documented in this
runbook structure (the only changes would be ┬ž3 provisioning and Tdarr
worker count back to 8).
